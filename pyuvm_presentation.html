Foundations

What is UVM?

  • UVM stands for Universal Verification Methodology.
  • It is the standard class-based methodology used to verify RTL designs.
  • Traditional UVM is built on top of SystemVerilog.
  • It promotes reusable components such as sequence items, sequences, drivers, monitors, agents, environments, and scoreboards.
  • The main idea is to verify at the transaction level instead of writing every test at the signal level.
  • That structure helps teams scale from a tiny ALU to a large SoC verification environment.

Simple Example

ALU transaction instead of raw pin toggling

Instead of writing separate signal assignments everywhere, UVM lets us describe a transaction like:

{ a: 7, b: 2, op: add }

The driver translates that transaction into DUT pin activity. The monitor turns DUT activity back into a transaction for checking.

Motivation

The Problem with Traditional UVM

  • Commercial simulator cost is often the first barrier for students, startups, and open-source projects.
  • SystemVerilog syntax is powerful, but it is also verbose and harder to learn for many software-oriented engineers.
  • Macros and boilerplate can make beginner UVM code feel heavier than the verification idea itself.
  • Open-source workflows are less natural in a traditional UVM setup.
  • Python ecosystems and CI tooling are easier to plug into than many classic simulator-centered flows.

Cost and Access

XTraditional UVM often depends on licensed simulators.
XLearning SystemVerilog + UVM together can slow onboarding.
OKPyUVM + cocotb + Icarus/Verilator/GHDL can be used in a zero-license flow.

Why that matters

Open-source verification lowers the barrier for teaching, prototyping, CI automation, and community-driven hardware development.

Core Technology

What is PyUVM?

  • PyUVM is a Python implementation of the UVM methodology.
  • It keeps the familiar ideas of components, phasing, sequences, TLM, config_db, and factory.
  • It uses cocotb to talk to the simulator and the DUT.
  • It lets verification engineers write UVM-style environments in Python instead of SystemVerilog.
  • That means easier scripting, faster iteration, simpler packaging, and access to the broader Python ecosystem.

Phases

What Are Phases? With Example

build
connect
run
check
report
  • Phases are ordered steps in the UVM testbench lifecycle.
  • build_phase creates components and sets configuration.
  • connect_phase wires ports, exports, or references together.
  • run_phase drives and monitors simulation-time activity.
  • check_phase evaluates correctness after run-time activity finishes.
  • report_phase prints final status and statistics.

Example from this ALU project

pythonclass AluEnv(uvm_env):

        self.agent = AluAgent("agent", self)
        self.scoreboard = AluScoreboard("scoreboard", self)

    def connect_phase(self):
        self.agent.monitor.scoreboard = self.scoreboard

        self.drop_objection()

        self.logger.info("ALU scoreboard checked")
Communication

What is TLM and How to Use It

  • TLM means Transaction-Level Modeling.
  • Instead of passing raw signals, components communicate using transactions.
  • That makes verification code more reusable and easier to reason about.
  • Common TLM usage includes sequencer to driver, monitor to scoreboard, and FIFO-style channels.
  • In PyUVM, TLM ports and exports are used to connect producers and consumers of transactions.

In this repo

The current code already uses the standard sequence-item handshake between sequencer and driver. The monitor then forwards completed items to the scoreboard with a direct Python call.

How to use it

pythonclass AluAgent(uvm_agent):
    def build_phase(self):
        self.seqr = uvm_sequencer("seqr", self)
        self.driver = AluDriver("driver", self)

    def connect_phase(self):
        self.driver.seq_item_port.connect(
            self.seqr.seq_item_export
        )

class AluDriver(uvm_driver):
    async def run_phase(self):
        while True:
            item = await self.seq_item_port.get_next_item()
            await bfm.send_op(item.a, item.b, item.op)
            self.seq_item_port.item_done()

Good mental model

Sequence creates the transaction. Sequencer hands it over. Driver consumes it. Monitor publishes observed transactions. Scoreboard checks them.

Customization

What is the Factory?

  • The factory is UVM's object creation mechanism.
  • It lets you create components and objects indirectly so you can replace them later without rewriting the environment.
  • This is useful for overrides, such as swapping in a tracing driver, error-injection driver, or alternative sequence.
  • In PyUVM, user classes are registered automatically, so there is no SystemVerilog-style registration macro burden.

Why engineers like it

You can keep the environment stable and change behavior from the test layer, which is cleaner than editing the environment every time.

Example override pattern

pythonclass LoggingDriver(AluDriver):
    async def run_phase(self):
        self.logger.info("Tracing transactions")
        await super().run_phase()

class FactoryOverrideTest(uvm_test):
    def build_phase(self):
        uvm_factory().set_type_override_by_type(
            AluDriver,
            LoggingDriver
        )
        self.env = AluEnv("env", self)

class AluAgent(uvm_agent):
    def build_phase(self):
        self.driver = AluDriver.create("driver", self)

Note: this repository currently instantiates the driver directly. The slide shows the factory-enabled pattern you would use when you want runtime overrides.

  • Sequence generates ALU operations such as add, subtract, multiply, and divide.
  • Sequencer arbitrates sequence items for the driver.
  • Driver converts each transaction into BFM calls.
  • BFM drives DUT inputs and samples DUT outputs through cocotb.
  • Monitor rebuilds observed transactions.
  • Scoreboard predicts expected outputs and checks actual results.
  • Test selects which sequence or set of sequences will run.

Project-specific detail

The divide sequence deliberately injects a divide-by-zero case so the scoreboard can verify both the result and the error flag.

Interface Layer

What is cocotb?

  • cocotb is a coroutine-based cosimulation framework for verifying HDL designs in Python.
  • It gives Python code access to simulator objects and DUT signals.
  • Testbenches use async/await and simulation triggers such as timers and clock edges.
  • It works with multiple simulators, which makes it a strong bridge between Python code and RTL simulation.

Plain-English definition

cocotb is the part that lets Python act like a verification language inside the simulator.

Small cocotb-style example

pythonfrom cocotb.triggers import Timer

async def send_op(self, a, b, op):
    self.dut.a.value = a
    self.dut.b.value = b
    self.dut.op.value = op
    await Timer(1, units="ns")

    observed = {
        "result": int(self.dut.result.value),
        "div_by_zero": int(self.dut.div_by_zero.value),
    }

That is exactly the style used by tb/alu_bfm.py in this repository.

  • PyUVM does methodology and organization.
  • cocotb does signal-level interaction with the simulator.
  • In this project, the driver never touches raw simulator APIs directly. It calls the AluBfm.
  • The BFM writes DUT inputs, waits for time to pass, then captures output values and queues them for the monitor.
  • This split keeps the testbench clean: PyUVM handles transactions while cocotb handles timing and signal access.

Tools

Open-Source RTL Simulation Tools

Icarus Verilog

Great for small Verilog/SystemVerilog teaching projects
  • Easy to install and widely used in examples
  • Works well with cocotb
  • Used by default in this repository's runner flow

Verilator

Fast compiled simulation for many Verilog designs
  • Often preferred when simulation speed matters
  • Strong fit for CI pipelines and large regressions
  • Works with cocotb for supported flows

GHDL

Open-source simulator for VHDL users
  • Useful when the DUT or environment is VHDL-centric
  • Lets the same Python methodology reach VHDL designs
  • Pairs well with cocotb in mixed toolchains

Waveform viewer

GTKWave is not a simulator, but it completes the debug loop by viewing VCD/FST waveform traces.

Practical advice

Pick the simulator based on HDL language support, speed needs, and the maturity of your specific DUT flow.

Next Steps

How to Extend This Example

  • Add more directed and random sequences to stress edge cases beyond simple arithmetic paths.
  • Track functional coverage for operations, corner values, sign behavior, and divide-by-zero scenarios.
  • Make the scoreboard more powerful by comparing against a reusable reference model.
  • Run regressions in CI so every RTL or testbench change is checked automatically.
  • Scale the same structure to a larger DUT by adding agents, interfaces, and more layered environments.
